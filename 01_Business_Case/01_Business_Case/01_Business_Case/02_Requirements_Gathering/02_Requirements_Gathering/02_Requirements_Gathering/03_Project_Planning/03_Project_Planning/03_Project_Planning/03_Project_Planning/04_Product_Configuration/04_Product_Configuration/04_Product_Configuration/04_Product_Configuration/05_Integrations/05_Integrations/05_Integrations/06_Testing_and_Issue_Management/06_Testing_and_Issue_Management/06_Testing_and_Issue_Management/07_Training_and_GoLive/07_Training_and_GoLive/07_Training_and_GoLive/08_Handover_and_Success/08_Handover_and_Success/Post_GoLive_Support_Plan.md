# Post-Go-Live Support Plan 📞

## Purpose

This document defines the post-go-live support approach for ABC GmbH after launching the corporate travel booking SaaS platform.

The goal is to support users during the first adoption period, monitor early issues, stabilize platform usage, and ensure a smooth transition from implementation to long-term account management.

## Support Period

The initial post-go-live support period covers the first 30 days after launch.

## Support Objectives

* Monitor first user logins
* Support first travel bookings
* Track approval workflow issues
* Monitor out-of-policy booking questions
* Support Finance reporting questions
* Resolve high-priority issues quickly
* Collect user feedback
* Prepare insights for the 30-day review

## Support Roles

| Role                   | Responsibility                                                                |
| ---------------------- | ----------------------------------------------------------------------------- |
| Implementation Manager | Supports launch stabilization, tracks issues, and coordinates early follow-up |
| Account Manager        | Owns long-term customer relationship and leads post-go-live review            |
| Customer Support Team  | Handles user support questions and recurring issues                           |
| IT Contact             | Supports login, SSO, and access-related issues                                |
| Travel Administrator   | Acts as first-line client-side support contact                                |
| Finance Manager        | Reviews reporting and cost center output                                      |
| Department Managers    | Support approval workflow adoption                                            |

## First Week Support Focus

During the first week after go-live, the team should monitor:

* Login success rate
* First employee bookings
* Approval workflow behavior
* Out-of-policy booking questions
* Cost center data accuracy
* Finance reporting access
* User support questions
* High-priority issues in the issue log

## Support Process

1. User reports an issue to the Travel Administrator or support contact.
2. Issue is logged with description, user type, priority, and impact.
3. Owner is assigned.
4. High-priority issues are escalated immediately.
5. Resolution is tracked in the issue log.
6. Recurring issues are reviewed for FAQ or training updates.
7. Open items are included in the 30-day review.

## Example Support Issues

| Issue                                      | Priority | Suggested Action                                    |
| ------------------------------------------ | -------- | --------------------------------------------------- |
| Employee cannot log in                     | High     | Check SSO and user access status                    |
| Manager does not receive approval request  | High     | Validate manager mapping and approval workflow      |
| Cost center missing from booking           | Medium   | Review employee data and cost center mapping        |
| Employee confused by out-of-policy warning | Medium   | Update FAQ with clearer example                     |
| Finance report missing required field      | Medium   | Review reporting requirement and escalate if needed |

## Communication Cadence

| Communication             | Frequency           | Owner                                    |
| ------------------------- | ------------------- | ---------------------------------------- |
| Go-live day update        | End of launch day   | Implementation Manager                   |
| First-week support review | Daily during Week 1 | Implementation Manager / Support Team    |
| Open issue review         | Weekly              | Implementation Manager / Account Manager |
| 30-day review             | After first month   | Account Manager                          |

## 30-Day Review Preparation

Before the 30-day review, collect:

* Number of active users
* Number of bookings
* Number of approval requests
* Common support questions
* Open issues
* Finance reporting feedback
* User feedback
* Improvement requests

## Success Indicators

Post-go-live support is successful when:

* Employees can access the platform
* First bookings are completed successfully
* Approval workflow works as expected
* Finance can access required reporting
* Critical issues are resolved quickly
* Account Manager has clear visibility into open follow-up items
* Customer feels supported after launch

## Business Value

A structured post-go-live support plan helps reduce adoption risk, improve customer confidence, and ensure the platform becomes part of the client’s daily travel process.
